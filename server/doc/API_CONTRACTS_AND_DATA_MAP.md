# API Contracts and Data Map

> Operational documentation for app-facing ManaLoom/mtgia API contracts. Evidence is from Dart Frog route handlers and Flutter providers in this repository. When a field or behavior was not fully proven from source, it is explicitly marked **not proven**.

## How Agents Must Use This File

- Consult this file before changing any app-facing endpoint, provider, DTO/model, sync job, cache behavior, or AI response consumed by the mobile app.
- Update this file in the same commit whenever a route, payload, response field, data source, tests/runtime evidence, or mobile consumer changes.
- Treat every listed response field as app-facing unless the endpoint status is `internal`.
- Preserve backward compatibility: newly added fields are optional unless a versioned contract and tests make them required.
- Do not remove or rename fields used by the listed app consumer files without updating the app and this document in the same change.
- Never document or return secrets: passwords, password hashes, JWT secrets, OpenAI keys, Scryfall credentials, database URLs, FCM tokens, user emails from real users, private keys, or raw Authorization headers.
- Avoid full schema dumps; document only app-facing fields and the main tables/columns needed to understand ownership and compatibility.
- Prefer source-code evidence over assumptions. If source evidence is incomplete, write **not proven** rather than guessing.
- When adding a route, add a contract row here with method/route, status, consumers, handler, request, response, tables/data source, tests/evidence, compatibility notes, and risks.

## Global Contract Rules

- **Authentication:** protected routes are behind middleware and read the authenticated user from `context.read<String>()` or `getUserId(context)`. App requests must send `Authorization: Bearer <JWT_REDACTED>`.
- **Database access:** handlers read `Pool` from `context.read<Pool>()`; mutations commonly use `pool.runTx()` for decks/trades and `pool.execute()` for direct CRUD.
- **Mobile data ownership:** mobile must **not** call Scryfall, MTGJSON, EDHREC, OpenAI, or other MTG/AI APIs directly. Backend owns external API usage, sync jobs, caches, calculations, validation, and AI orchestration.
- **Response format:** handlers use `Response.json(body: {...})` or return JSON arrays for some legacy routes (`GET /decks`). Errors commonly use `{error: ...}` or `{message: ...}`.
- **Pagination:** app-facing list routes use `page` and `limit` query params. Limits are capped in handlers (commonly 50, 100, or 200). Response shapes are usually `{data, page, limit, total}` or `{data, page, limit, total_returned}`.
- **Optional fields:** `archetype`, `bracket`, `pricing_*`, set metadata, trust metrics, price insights, telemetry, AI diagnostics, and runtime analysis fields are optional/evolving unless listed as required in a request body.
- **Route status vocabulary:** `stable` = app-facing and currently consumed; `experimental` = app-facing but AI/telemetry/simulation or evolving; `internal` = operational/admin/diagnostic; `deprecated` = retained only for compatibility; `not proven` = file exists or consumer exists but source evidence was incomplete.
- **Caching:** `EndpointCache.instance` is used by cards and sets list-style queries (`/cards` 45s TTL; `/sets` cache present in route). Optimize uses persistent `ai_optimize_cache` with cache metadata in response.
- **Validation:** deck legality and construction rules are backend-owned through `DeckRulesService`; app sends deck/card intent, backend validates.
- **Security:** no secrets/tokens/password hashes in responses. `/users/me/fcm-token` stores/removes FCM tokens but returns only `{ok: true}`.
- **CORS:** browser origins are exact-match allowlisted by `MANALOOM_ALLOWED_ORIGINS`. Production never falls back to `*` or localhost. Loopback is accepted only outside production with `MANALOOM_ALLOW_DEV_ORIGINS=true`; native clients without an `Origin` header remain supported.
- **Deleted accounts:** `users.deleted_at` is authoritative. Login, token lookup and protected middleware reject deleted accounts, immediately revoking previously issued JWTs after anonymization. Deleted accounts and their decks/binders are excluded from community search, public discovery, marketplace and follow targets. Migration 038 installs active-user FK guards that serialize every user-linked insert/update with account deletion; route-level locks additionally validate multi-party social writes. Retained history may show only the pseudonymous deleted-user row.
- **Compatibility:** existing app code expects permissive optional parsing. Do not make optional response fields mandatory without app coordination.

#### Async AI job lifecycle v2 (migration 048)

The following lifecycle contract is authoritative for both generation and
optimization jobs and supersedes the narrower status descriptions in the table
rows above:

- `POST /ai/generate` and `POST /ai/optimize` accept optional `request_key`, a
  trimmed 1–128 character key restricted to letters, digits, `.`, `_`, `:`, and
  `-`. The app creates one key per player action and reuses it for retry/resume.
  The same authenticated user and key return the original job without starting
  another worker or reserving quota again. Reusing the key with different
  canonical request material returns `409` with
  `error_code=ai_job_idempotency_conflict`.
- Every async `202` includes `poll_url`, `cancel_url`, `resume_url`, bounded
  polling/timeout hints, and
  `idempotency.{request_key,reused}`. A reused request can return a job already
  in progress or terminal; clients must inspect the job status.
- `GET /ai/generate/jobs/:id` and `GET /ai/optimize/jobs/:id` return the closed
  status set `pending`, `processing`, `completed`, `failed`, `cancelled`, plus
  `progress.{current,total,ratio}`, `can_cancel`, `can_resume`, `poll_url`, and
  `cancel_url`. Optimize jobs also return `deck_id` and `archetype`.
- `GET /ai/generate/jobs/latest?active=true` returns the current user's latest
  nonterminal generation job. `GET /ai/optimize/jobs/latest?active=true&deck_id=`
  does the same for one owned deck. No row, expired row, or another user's row
  returns `404`.
- `DELETE /ai/generate/jobs/:id` and `DELETE /ai/optimize/jobs/:id` atomically
  change an owned `pending`/`processing` job to `cancelled`. Missing/cross-user
  ids return `404`; terminal jobs return `409`; `DELETE .../latest` returns
  `405`.
- All worker progress, completion, and failure writes are conditional on the
  current status still being `pending`/`processing`. A cancelled or timed-out
  job therefore cannot be resurrected by a late worker. Generate stalls fail
  after four minutes and optimize stalls after six minutes.
- `ai_generate_jobs` and `ai_optimize_jobs` persist `request_key`,
  `request_fingerprint`, and `cancelled_at`; each has a partial unique index on
  `(user_id, request_key)` and a status constraint that includes `cancelled`.
  `/health/ready` verifies migration 048, both constraints, all six columns,
  and both indexes before reporting ready.

## Known Risks

- **AI contracts are intentionally experimental.** `/ai/optimize`, `/ai/generate`, `/ai/rebuild`, simulations, commander reference, and AI analysis depend on backend heuristics, DB state, cache, and sometimes OpenAI/EDHREC/MTGTop8-style sources. App code must parse defensively.
- **Async optimize split:** `/ai/optimize` can return sync 200 or async 202. Consumers must poll `/ai/optimize/jobs/:id` and handle failure/expiration.
- **Schema-aware optional fields:** deck meta/pricing fields may be null when migrations are absent or when helpers fall back. Do not assume fields are non-null.
- **Large payloads:** deck details, public deck details, and optimize responses can include 100-card Commander lists plus analysis; watch mobile performance.
- **Cache staleness:** cards/sets use short TTL in-memory cache; optimize uses persistent cache. After changing analyzer or optimization logic, clear `ai_optimize_cache` before validation.
- **External dependency failures:** OpenAI key absence may return explicit mock responses only in non-production profiles. Production fails closed, reports the provider as unavailable in readiness, and never presents mock output as provider output. External MTG source failures may degrade through audited backend fallback/cache. Mobile must never bypass backend.
- **Trust/price sparse data:** marketplace and trade trust/value fields can be incomplete for new users or cards with missing `price_history`/price data.
- **Marketplace/trades trust shape:** marketplace trust currently lives at `owner.trust`; trade trust lives at `sender.trust`/`receiver.trust`; trade `value_summary` currently lives only on `GET /trades/:id`, not on `GET /trades`.
- **Incomplete route proof:** Some existing routes are documented as **not proven** for exact field shapes. Before relying on those fields, inspect handler and add contract tests.
- **Legacy shapes:** `GET /decks` returns a raw JSON array while most newer lists return `{data, page, limit, total}`.
- **No secrets in docs:** The repository contains local environment/config files; never copy credentials into this or any other committed doc.

## Update Checklist

When changing an app-facing contract:

1. Identify every app consumer file and backend handler affected.
2. Update this document in the same change with method/route, status, request, response, optional fields, data source, tests, and risks.
3. Keep new response fields optional unless a versioned migration is coordinated.
4. Do not remove/rename fields consumed by the app without updating the app provider/models and tests.
5. Add or update server contract tests for backend behavior.
6. Add or update app integration/runtime evidence for user-visible flows.
7. Confirm mobile still calls backend only for MTG/AI data; do not introduce direct external MTG API calls.
8. Verify no secrets, JWTs, password hashes, FCM tokens, API keys, or database URLs are documented or returned.
9. For AI/optimization changes, clear `ai_optimize_cache` and run the relevant optimize/generation tests or quality gate.
10. Run `git diff --check` and a targeted test/gate appropriate to the changed module.
