# ManaLoom Hermes Operating Protocol

This repository is `softwarePredador/mtgia`. Follow this protocol before any
implementation, verification, commit, or push.

## Non-Negotiable Git Rules

1. Always start with:
   ```bash
   pwd
   git rev-parse --show-toplevel
   git branch --show-current
   git status --short --branch
   ```
2. Do not change branches if `git status --short` is not clean, unless the user
   explicitly asked you to preserve or discard the visible changes.
3. Never use low-level Git workarounds for normal development:
   - no `git hash-object`
   - no `git update-index`
   - no `git write-tree`
   - no `git commit-tree`
   - no manual edits under `.git/objects`, `.git/refs`, or `.git/index`
4. If `git add`, `git commit`, `git diff`, or `git push` fails, stop and report
   the exact failure. Do not invent a workaround.
5. Before committing, run:
   ```bash
   git diff --check
   git status --short
   ```
6. Commit only files that are part of the requested task. Explain every
   untracked file before deciding whether to add it.
7. Never commit local databases, generated scratch files, secrets, raw dashboard
   HTML, `.env`, credentials, API keys, PATs, JWTs, DSNs, or private payloads.

## Implementation Rules

1. Read the relevant project sources before changing code:
   - `.github/instructions/guia.instructions.md`
   - `docs/CONTEXTO_PRODUTO_ATUAL.md`
   - `server/manual-de-instrucao.md`
   - `server/doc/API_CONTRACTS_AND_DATA_MAP.md`
   - any `.github/agents/*.agent.md` file that matches the task domain.
2. Use the smallest safe change. Do not start broad refactors while fixing a
   focused bug.
3. If extracting code into a new file, the same commit must also wire imports,
   remove duplicated old definitions, and run analyzer/tests. Do not leave
   untracked extraction files as "future work".
4. For app-facing backend contract changes, update
   `server/doc/API_CONTRACTS_AND_DATA_MAP.md` and `server/manual-de-instrucao.md`
   unless the change is strictly internal and backward-compatible.
5. Mark uncertain findings as `not_proven`; do not present guesses as facts.

## Required Validation

For backend Dart changes, always run at least:

```bash
cd server
dart analyze
dart test <focused_test_files>
```

For optimize/AI route changes, include the relevant focused tests, usually:

```bash
cd server
dart test test/optimization_validator_test.dart \
  test/ai_optimize_semantic_enforcement_route_contract_test.dart
```

If a validation command cannot run because of environment limits, report the
exact command, the exact failure, and what remains unproven.

## Final Response Checklist

Every implementation response must include:

- branch name;
- commit hash if committed;
- files changed;
- validation commands and results;
- untracked files left behind, if any;
- explicit statement if no push was performed.
